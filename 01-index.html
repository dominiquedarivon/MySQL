SGBDR ou SGBD

  • Le logiciel client permet de dialoguer avec le serveur grâce au langage SQL

  • Concurrens: Oracle, MS Access SQLite, PostgreSQL
  • SGBDR :Système de gestion de base de données relationnelles
  • SGBD : Système de gestion de base de données

En résumé

  • My SQL est un SGBDR, un système de gestion de base de données
  • Le langage SQL est utilisé pour la communication entre le client et le serveur.
  • Dans une BDD, les données sont représnetées sous forme de tables.
  • Dans une table, nous avons des colonnes, chaque nouvelle donnée sera introduite sous forme de ligne dans le table

1- Modélisation d'une base de donnée BDD

Avant de créer notre base de données, il est essentiel de se poser et de réfléchir à la modélisation. Rien est irréversible, mais changer la modélisation en cours de projet est déconseillée car cela forcera certainement à réecrire une partie du code ce qui peut prendre beaucoup de temps et avoir des conséquences sur ce qu'on à déjà fait comme des dérèglemebts d'autres parties de l'application . C'est la raison pour laquelle il vaut mieux éviter de se tromper.

Comment modéliser une BDD

Si la BDD n'est pas complexe, il est possible de la modéliser directement avec une feuille et un stylo. Cependant dans la plupart des cas , nous aurons besoin d'un logicielpour accompagner et avoir une vue d'ensemble. Un logiciel connu permet de modéliser une BDD, c'est MySQL workbench. Il existe cependant de nombreux outils permettant cette manipulation. Pour une bonne conception de sa base de données il faut réfléchir en terme de sujet en modélisant ceux du monde réel. Par exemple , si nous vendons des produits sur notre site; il y aura une table utilisateur, une table produits et une table commande(cette table rassemblera l'id utilisateur et celui du produit ).

2- Les tables

  • Un sujet représente une table dans une BDD
  • une table est un emplacement de sauvegarde
  • Les données sont stockées à l'intérieure de tables. Une table peut être comparée à une liste qui contient des enregistrements relatfs à un sujet bien défini

Il faut en fait , dès le départ , réfléchir à chaque sujet abordé dans notre site et à ses conséquences.Si nous vendons , nous aurons certainement des membres. Une table membre ( user) sera donc nécessaire . En terme de fonctionnalité, si nous souhaitons proposer un abonnement newsletter, nous arons besoin d'une table newsletter qui sera liée à la table meme(user). 1 table = 1 sujet

3- Les colonnes ou les champs

chaque table possède généralement plusieurs champs, aussi appelé colonnes. Les colonnes représentent des caractéristiques relatives au sujet (-> la table).

Pour savoir quelle colonne mettre dans quelle table, il faut se poser les questions suivantes : qu' est ce qui pourrait décrire mon sujet? quelles sont les informations sur mon sujet?
La table user aura par exemple les champs : pseudo, mot de passe, ,om , prénom etc.

4-Les types de données

Type de champs Description Contexte

5-Types de données, PK,AI pour les identifiants

a) Les identifiants, PK(Primary Key - Clé primaire)

  • Les identifiants sont des champs un peu particuliers car c'est la seule colonne qui ne décrit pas le sujet ? Ils sont cependant systematiquement présents dans chaque table en première position
  • chaque table possède une colonne identifiant que l'on appelera la clé primaire ou tout simplement PK.
  • On pourra tout simplement appeler "id" mais on retrouvera parfois 'idProduit' ou encore 'idMembre', etc.

b) AI, l'auto incrémentation

Auto_Increment est une option permettant de générer un numéro unique dans une colonne de type clé primaire (PK).Par consequent, retenez bien que le premier champ de chaque table sera systèmatiquement un "id" qui sera à la fois (PK)(Primary key) et AI(Auto_increment).

c) NULL / NOT NULL

Dans chaque champ, nous pourrons indiquer si nous acceptons les valeurs NULL ou NOT NULL. Null est un type de valeur en informatique permettant de laisser un champ vide si nous n'avons pas d' information à y laisser.

d)Les relations

Il arrive parfois que les sujets interagissent entre eux

  • Exemple : 1 "membre" commande 1 "produit", nous devons enregistrer l'information dans une table"commande".
  • Autre exemple: société UBER Un conducteur conduit un véhicule . Il faudra une table permettant de préciser quel conducteur va avec quel véhicule .

6- Les cardinalités

Les cardinalités permettent de connaître le chiffre minimum et maximum d'enregistrements pour une relation.

Par exemple, un enfant est la descendance d'une seule mère biologique (minimum = 1/ maximum = 1).En revanche, une femme peut avoir plusieurs ou aucun (minimum=0 /maximum = n).c'est à dire infini

Autre exemple, 1 livre possède forcément 1 auteur et 1 max (1,1).

7- Table de jointure

Une table de jointure permet de faire le lien entre deux tables. Par exemple, prenons le cas d'une société de taxis qui posséderait des conducteurs(table conducteurs) et des véhicules (table véhicules).

  • Un conducteur peut conduire plusieurs véhicules
  • Un véhicule peut être conduit par 0 ou plusieurs conducteurs.

    • Comment savoir quel conducteur conduit quel véhicule ?

    Grace à la table de jonture , nous savons qu'Alexandre conduit à la fois une mercede et la BMW et que julien ne que conduit que la BMW.

Les clés étrangères , Foreign KEY (FK)

Les clés étrangères sont des champs forcément clé primaire dans leur table d'origine et qui sont présents dans une table extéreure.
Un champ clé étrangère (FK) placé à l'extérieur de sa table d'origine permet de la relation avec un autre sujet

9- Les enregistrements

Chaque enregistrement représente une ligne dans la BDD et chaque enregistrement est auto incrémenté. Même si un enregistrement est sypprimé, son id ne sera pas réutilisé.

10- Comment interroger la BDD ?

a) Les requêtes

Une requête permet de poser une question, d'interroger la base de donnée (BDD) ou encore de donner un ordre (CRUD -> Creat = créer / Read= lis une infor/ Update = met à jour/ Delet = Supprime)

Nous terminerons toutes nos requêtes SQL par un point virgule , pour signifier qu'il faut exécuter la demande.

Types de requêtes

Il existe 4 types de requêtes

  • La sélection : SELECT - c'est une question / réponse => nous faisons une demande et obtenons une réponse
  • L'insertion : INSERT : requête d'action-> impact sur les données
  • La modification : UPDATE - requête d'action -> on modifie les données déjà présentes dans notre table.
  • La suppression : DELETE - requête d'action -> on supprime un enregistrement de notre table // une table // une BDD
